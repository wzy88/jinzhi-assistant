# 下一步补齐清单

这份清单区分“代码已补齐”和“需要外部配合”。代码侧可以继续迭代；真实试点证据需要项目负责人提供材料。

## 代码侧已补齐

- 前端单条诉求入口：粘贴诉求后可调用 `/api/analyze`，失败自动走本地规则。
- 通知生成入口：优先调用 `/api/notice`，失败自动走本地模板。
- 准确率口径：未人工复核前显示“待评测”，不再默认显示高准确率。
- 批量导入：跳过 `sample_tickets.csv` 表头，避免把表头当工单。
- 工具箱弹窗：四个弹窗已中文化并补齐基础交互。
- 演示/真实数据切换：关闭“显示演示样例数据”后只保留导入或单条分析生成的记录。
- 试用反馈留痕：试点分析页可记录试用者、角色、评分和卡点，并导出反馈 CSV。
- 评测证据导出：试点分析页展示已复核、正确、需调整、待复核，并导出评测明细 CSV。
- 自动化测试：新增 Playwright 冒烟测试，覆盖登录、单条分析和工具箱弹窗。

## 需要你配合提供

- DashScope API Key：用于验证线上云端模型调用，而不是只走本地兜底。
- DeepSeek API Key：如果你想把默认云端模型切到 DeepSeek，也可以直接用它。
- 10-30 条真实脱敏诉求：至少包含人工类别、人工紧急度、来源和备注。
- 1-3 位试用对象：社区网格员、物业客服或高校后勤人员均可。
- UAT 记录：每位试用者完成一次工单处理、通知生成、复核和周报查看，并记录耗时和卡点。
- 线上部署地址：Vercel 项目 URL 和 `jinzhi.22dhmv.top` 绑定状态。
- 比赛材料素材：1 分钟演示视频、产品截图、访谈截图或试用意向证明。

## 下一轮建议

1. 用真实脱敏样例跑准确率评测。
2. 用 1-3 位真实试用者填写 UAT 反馈，并导出反馈 CSV 放入比赛材料。
3. 接服务端数据库，保存试点工单、复核记录和反馈记录。
4. 补移动端截图和 Playwright 移动端测试。
5. 生成可提交的 PDF/图表证据包。
