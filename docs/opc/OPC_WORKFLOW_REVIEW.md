# OPC 工作流复盘

## 1. 复盘目标

这份文档用于展示：一个人如何借助 AI，完成传统团队中产品经理、设计师、工程师、测试、交付、销售和运营的工作。

## 2. 一人多角色拆解

| 传统角色 | 本项目中的工作 | AI 协作方式 |
| --- | --- | --- |
| 市场研究 | 阅读比赛文档，识别智慧生态和社区网格机会 | 提炼赛道要求、评审权重和命题关键词 |
| 产品经理 | 写 PRD、需求池、优先级和验收标准 | 生成用户故事、MVP 范围、风险表 |
| UX 设计师 | 设计单页工作台、核心流程和交互状态 | 生成信息架构、页面结构、交互流程 |
| UI 设计师 | 制定克制可信的政务工具风格 | 生成视觉原则、布局建议和响应式要求 |
| 前端工程师 | 实现 React 工作台 | 辅助编写组件、状态管理和样式 |
| 后端工程师 | 实现 Vercel API 和模型调用 | 辅助设计接口、JSON 标准化和兜底逻辑 |
| 测试人员 | 制定测试计划和冒烟用例 | 生成测试矩阵、准确率评测方法 |
| 交付经理 | 制作社区使用 SOP 和培训流程 | 生成交付手册、回访问题、验收标准 |
| 销售顾问 | 设计免费试点、订阅、项目交付 | 生成价格包、销售话术和转化路径 |
| 运营复盘 | 整理比赛证据链和下一步计划 | 生成路演主线、证据清单、复盘材料 |

## 3. 传统方式 vs OPC 方式

| 环节 | 传统小团队 | OPC + AI |
| --- | --- | --- |
| 立项分析 | 1-2 天多人讨论 | 半天内完成赛道拆解和项目定位 |
| PRD | 产品经理 1-2 天 | AI 辅助 2-4 小时形成初稿 |
| 设计 | 设计师 1-3 天 | 先用文字设计规格 + 代码实现验证 |
| 开发 | 前后端 3-7 天 | 单人完成前端、API 和部署准备 |
| 测试 | 测试人员写用例 | AI 生成测试矩阵，单人执行 |
| 交付材料 | 运营/交付整理 | AI 辅助快速生成 SOP 和话术 |
| 商业计划 | 创始人/销售打磨 | AI 辅助拆价格、客户和转化路径 |

## 4. 人效提升假设

| 指标 | 传统估计 | 本项目目标 |
| --- | --- | --- |
| MVP 原型 | 1-2 周 | 1-3 天 |
| 产品文档 | 2-3 天 | 0.5-1 天 |
| 测试计划 | 1 天 | 2 小时 |
| 交付 SOP | 1 天 | 2 小时 |
| 商业方案 | 1-2 天 | 0.5 天 |

这些数据在决赛前需要用实际工作记录进一步校准。

## 5. AI 使用边界

AI 负责：

- 文档初稿。
- 需求梳理。
- 代码辅助。
- 测试用例设计。
- 商业方案草拟。
- 试点材料整理。

人负责：

- 判断真实用户痛点。
- 选择优先级。
- 审核模型输出。
- 联系试点对象。
- 做最终交付承诺。
- 承担数据和隐私责任。

## 8. 路演表达

不要只说“我做了一个社区 AI 工具”，要说：

> 我用 OPC 的方式，把一个政务场景机会从立项、需求、设计、开发、测试、交付、商业化全部跑了一遍。津智助理是产品本身，而这套项目包证明了它不是灵感 Demo，而是一套可以继续试点和复制的单人公司交付方法。
