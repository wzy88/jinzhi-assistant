# 社区交付手册

## 1. 交付目标

让社区、物业或后勤人员不用安装软件，只通过一个网页链接，就能完成居民诉求预处理、通知生成、人工复核和周报草稿生成。

## 2. 交付对象

| 角色 | 使用内容 |
| --- | --- |
| 网格员 | 单条工单分析、通知生成、处理状态 |
| 物业客服 | 批量导入、分类、分派建议 |
| 社区负责人 | 高频问题看板、周报草稿、反馈评估 |
| 试点联系人 | 数据脱敏、反馈表、试用总结 |

## 3. 交付前准备

- [ ] 产品已部署到公网。
- [ ] 已确认是否使用云端模型或规则引擎。
- [ ] 已准备 10-30 条脱敏工单样例。
- [ ] 已告知用户不要输入姓名、电话、身份证、具体门牌号等隐私。
- [ ] 已准备 15 分钟培训话术。
- [ ] 已准备试用反馈表。

## 4. 社区首次使用 SOP

### 第一步：打开产品

访问：

```text
https://jinzhi.22dhmv.top
```

如果线上地址暂未部署，可使用本地演示地址：

```text
http://127.0.0.1:5173/
```

### 第二步：输入诉求

把脱敏后的居民诉求粘贴到“居民诉求文本”。

示例：

```text
12号楼消防通道长期被私家车占用，晚上救护车进不来，希望社区协调。
```

### 第三步：分析并入库

点击“智能分析并入库”，查看：

- 诉求类别
- 紧急程度
- 情绪标签
- 建议部门
- 工单摘要
- 建议动作

### 第四步：人工复核

如果结果基本正确，点击“判断正确”。

如果结果不符合实际，点击“需调整”，并在反馈表中记录原因。

### 第五步：更新处理状态

根据实际进度选择：

- 待转派
- 已转派
- 已回访

### 第六步：生成通知

输入通知主题，点击“生成通知”，复制适合的版本：

- 正式版：公告或正式通知。
- 居民群版：微信群。
- 短信版：短消息或简短提醒。

### 第七步：查看周报

一批工单处理后，进入“试点看板”，刷新周报草稿，复制到工作汇报材料。

## 5. 15 分钟培训流程

| 时间 | 内容 |
| --- | --- |
| 0-2 分钟 | 介绍项目用途和隐私边界 |
| 2-5 分钟 | 演示单条消防通道诉求 |
| 5-8 分钟 | 演示通知生成和复制 |
| 8-11 分钟 | 演示处理记录、复核、状态流转 |
| 11-13 分钟 | 演示看板和周报 |
| 13-15 分钟 | 收集问题和试用意向 |

## 6. 试点运行方式

### 轻量试点

- 周期：1-3 天。
- 数据：10-30 条脱敏样例。
- 目标：验证分类、通知、周报是否有用。

### 标准试点

- 周期：1-2 周。
- 数据：50 条以上脱敏工单。
- 目标：形成准确率、节省时间、用户反馈报告。

### 深度试点

- 周期：1 个月。
- 数据：接入服务端数据库。
- 目标：验证付费意向和交付成本。

## 7. 数据安全要求

1. 不收集居民姓名、电话、身份证。
2. 楼栋房号可以模糊成“某楼栋”“某单元”。
3. 试点截图公开前必须二次脱敏。
4. AI 输出只作为辅助建议，不替代人工判断。
5. 所有比赛材料中使用的数据必须标注“脱敏”或“模拟”。

## 8. 交付验收

| 项目 | 验收标准 |
| --- | --- |
| 产品可访问 | 社区人员能打开链接 |
| 核心流程 | 能独立完成 1 条诉求分析 |
| 通知生成 | 至少 1 个版本可复制使用 |
| 复核记录 | 能标记正确/需调整 |
| 看板周报 | 能生成一份周报草稿 |
| 反馈收集 | 至少填写 1 份试用反馈 |

## 9. 交付后回访

回访问题：

1. 哪个功能最有用？
2. 哪个字段最不可信？
3. 是否真的节省时间？
4. 是否愿意继续免费试用？
5. 如果要正式使用，必须补什么？
6. 是否愿意推荐给其他社区/物业？

## 10. 试点转化下一步

如果反馈正向，推进：

1. 继续免费试用 1-3 个月。
2. 补服务端持久化和账号权限。
3. 输出试点报告。
4. 提供社区/物业报价单。
5. 寻找街道、园区、物业公司的批量试点机会。
