# 需求规格说明

## 1. 用户故事

### 网格员

作为社区网格员，我希望把居民诉求粘贴到网页后，系统自动给出类别、紧急程度、情绪、建议部门和处理步骤，这样我可以更快判断应该先处理什么、转给谁。

### 物业客服

作为物业客服，我希望批量导入报修或投诉文本，系统帮我归类和生成摘要，这样我可以快速分派给保洁、工程或客服人员。

### 社区负责人

作为社区负责人，我希望看到本周高频问题、复核准确率和周报草稿，这样我可以判断社区治理的重点问题。

### 评委

作为比赛评委，我希望看到一个真实可访问、可演示、可试点、可变现的工具，而不是停留在概念页。

## 3. MVP 验收标准

| 功能 | 验收标准 |
| --- | --- |
| 工单分析 | 输入消防、卫生、维修、邻里纠纷样例，能输出合理分类 |
| 紧急度判断 | 消防通道、电动车充电等风险类诉求应为高优先级 |
| 通知生成 | 能输出正式、居民群、短信三种可复制文本 |
| 处理记录 | 每次分析后记录增加，能回看原始诉求和分析结果 |
| 人工复核 | 点击正确/需调整后，准确率指标更新 |
| 批量导入 | 导入 CSV/TXT 后生成多条记录 |
| 周报 | 能基于当前记录生成统计摘要和建议动作 |
| 演示稳定性 | 无 API Key 时仍能使用规则引擎完成演示 |

## 4. 数据字段

### 工单输入

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| text | string | 居民诉求文本 |
| source | string | 来源，初赛可选：微信/电话/线下/导入 |
| createdAt | datetime | 创建时间 |

### AI 输出

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| category | string | 诉求类别 |
| urgency | enum | 高/中/低 |
| emotion | string | 情绪标签 |
| department | string | 建议处理部门 |
| summary | string | 工单摘要 |
| nextSteps | string[] | 建议动作，最多 3 条 |
| confidence | number | 置信度 |

### 人工复核

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| review | enum | 待复核/正确/需调整 |
| status | enum | 待转派/已转派/已回访 |
| humanCategory | string | 人工标注类别，评测用 |
| humanUrgency | string | 人工标注紧急度，评测用 |
| feedback | string | 使用者反馈 |

## 5. 明确不做

初赛不做完整政务审批流、不接入真实隐私数据、不自动发短信、不做多租户复杂权限、不替代人工最终判断。

## 6. 下一轮必须补齐

1. 让“显示演示样例数据”真正控制样例数据显隐。
2. 增加试用反馈收集入口。
3. 补 30 条样例数据的准确率评测。
4. 用真实脱敏样例压测自动脱敏规则。
5. 为真实试点补服务端持久化和权限边界。
